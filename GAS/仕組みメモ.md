# CAFE SHIFT GAS 仕組みメモ

## 全体像

- `GAS/コード.gs`
  - サーバー側の本体。
  - Webアプリの入口、認証、スプレッドシート操作、通知、AI下書き作成を担当。
- `GAS/AdminView.html`
  - 管理者画面。
  - 募集作成、必要人数設定、AI下書き、手動調整、確定公開、スタッフ管理を担当。
- `GAS/StaffView.html`
  - スタッフ画面。
  - ログイン、登録、希望提出、変更申請、欠員への立候補、プロフィール更新を担当。

## 画面の入口

- `doPost(e)`
  - Google IDトークンを検証して、短命のセッショントークンを発行するAPI。
  - `createSession` と `ping` に対応。
- `doGet(e)`
  - `st` パラメータのセッショントークンを確認。
  - `view=admin` のとき `AdminView`、それ以外は `StaffView` を返す。
  - つまり表示URLはセッション前提。

## データの中心

- ベーススプレッドシート: `BASE_SS_ID`
  - スタッフ一覧
  - 募集履歴一覧
  - お知らせ
- 募集ごとの個別スプレッドシート
  - `TEMPLATE_SS_ID` をコピーして作成。
  - `基本設定`
  - `提出情報`
  - `必要枠`

## `コード.gs` の役割整理

### 1. 認証とセッション

- `_verifyGoogleIdToken_(idToken)`
  - Googleの `tokeninfo` でIDトークンを検証。
- `_isAllowedEmail_(email)`
  - `ALLOWED_EMAILS` に含まれるメールのみ許可。
- `_createSession_(email)` / `_getSession_(token)`
  - `CacheService` に1時間のセッションを保存。

### 2. 募集データの取得と整形

- `getRecruitSS(targetId)`
  - `募集履歴一覧` から対象募集のスプレッドシートを引く。
- `parseRecruitSettings(setVals)`
  - `基本設定` から期間、締切、営業時間、例外営業日、時間帯一覧を作る。
- `getReqMap(recruitSs)`
  - `必要枠` を `日付+時間` 単位のマップに整形。
- `getRecruitListsForAdmin()`
  - 募集一覧を `all/open/edit/confirmed` に分けて返す。

### 3. スタッフ向け機能

- `verifyStaffPin(name, pin)`
  - スタッフ一覧の名前+PINでログイン。
- `registerStaff(req)`
  - 新規スタッフを `申請中` で追加。
- `updateStaffProfile(oldName, req)`
  - スタッフ自身の設定更新。
- `getStaffInitData(staffName, recruitId)`
  - スタッフ画面の初期表示データをまとめて返す。
- `submitShiftData(recruitId, name, slots)`
  - 希望シフト提出。
- `sendChangeRequest(recruitId, dt, name, reason)`
  - 確定済みシフトへの変更申請。
- `applyForVacancy(recruitId, dt, name)`
  - 欠員枠へ立候補。

### 4. 管理者向け機能

- `getAdminInitData(recruitId)`
  - 管理者画面の初期表示データをまとめて返す。
- `startNewRecruit(data)`
  - テンプレートのシートを複製して新規募集を作成。
- `saveReqSlots(recruitId, slots)`
  - 必要人数を `必要枠` シートに保存。
- `updateAdminStaffStatus(name, status, role, note)`
  - スタッフ情報の状態やメモを更新。
- `executeShiftAction(...)`
  - 変更申請の承認/却下、立候補承認、手動追加・削除・入替をまとめて処理。
- `publishRecruit(recruitId)`
  - 募集履歴を `確定済` にし、`仮決定` を `確定` に変える。
- `sendRemind(staffName)`
  - 未提出者への催促通知。

### 5. AI補助

- `extractStaffPrefsByGemini(note)`
  - スタッフ備考から希望条件をGeminiで構造化。
- `generateShiftByAI(options)`
  - `必要枠` と `提出情報` とスタッフ情報を使って `仮決定` を作る。
  - 目安時間や希望条件を見ながら割当している。

## `AdminView.html` の流れ

### 初期表示

- `window.onload -> loadData(null)`
- `loadData()` は `getAdminInitData()` を呼ぶ。
- 受け取ったデータで以下を描画する。
  - ホームのアラート
  - 今日の確定シフト
  - 募集セレクタ
  - 必要人数表
  - AI編集表
  - 確定/申請対応表
  - スタッフ一覧

### 主な操作と呼び先

- 新規募集開始
  - `confirmStartRecruit()` -> `startNewRecruit(data)`
- 必要人数保存
  - `executeAIGeneration()` の最初で `saveReqSlots(...)`
- AI下書き作成
  - `executeAIGeneration()` -> `generateShiftByAI({ recruitId })`
- 公開
  - `runPublishAI()` -> `publishRecruit(currentRecruitId)`
- スタッフ更新
  - `executeStaffEdit(name)` -> `updateAdminStaffStatus(...)`
- 承認/却下/手動調整
  - `runAction()`, `runVolAction()`, `executeManual()`
  - いずれも最終的に `executeShiftAction(...)`

### 画面上の意味

- `s-new`: 募集作成
- `s-req`: 提出状況確認、必要人数設定
- `s-edit`: AI下書きと手動調整
- `s-conf`: 変更申請、立候補対応

## `StaffView.html` の流れ

### 初期状態

- まずログイン画面を表示。
- ログイン成功後に
  - `verifyStaffPin(name, pin)`
  - `loadInitData(currentRecruitId)` -> `getStaffInitData(myName, recruitId)`

### 主な操作と呼び先

- 新規登録
  - `confirmRegister()` -> `registerStaff(data)`
- 希望提出
  - `executeSubmit()` -> `submitShiftData(currentRecruitId, myName, slots)`
- 欠員立候補
  - `applyVacancy(dt)` -> `applyForVacancy(currentRecruitId, dt, myName)`
- 変更申請
  - `executeChangeRequest(dt)` -> `sendChangeRequest(currentRecruitId, dt, myName, reason)`
- プロフィール更新
  - `executeProfileUpdate()` -> `updateStaffProfile(myName, data)`

### 表示内容

- ホーム
  - お知らせ
  - 今日から1週間の確定シフト
  - 募集中アラート
- シフト
  - 募集中: 希望提出表
  - 提出内容: 自分の提出済み内容
  - 確定版: 全スタッフの確定表
- 設定
  - 自分の基本情報更新

## 状態管理の見方

- `提出情報` の1列目ステータスが重要。
- 主に使われている値
  - `希望`
  - `仮決定`
  - `確定`
  - `変更申請`
  - `立候補`

このステータスで、管理画面とスタッフ画面の見え方が切り替わる。

## 実装を触るときの着眼点

- 画面の表示崩れやボタン挙動は、まず各HTML内の `google.script.run` 呼び先を見る。
- データ不整合は、`コード.gs` の
  - `getAdminInitData`
  - `getStaffInitData`
  - `executeShiftAction`
  - `generateShiftByAI`
  を追うと全体像が掴みやすい。
- 募集の時間帯表示は `parseRecruitSettings()` が起点。
- 必要人数の保存先は `必要枠`、実際の提出や確定は `提出情報`。

## 今の理解メモ

- これは「1つのベース台帳 + 募集ごとの個別シート」で動く構成。
- 管理画面は一覧・編集・承認のハブ。
- スタッフ画面は個人起点で、提出と確認に絞っている。
- 認証は二系統ある。
  - Webアプリ入口は Google IDトークンベースのセッション
  - スタッフ画面の通常利用は 名前+PIN
